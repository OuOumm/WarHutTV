## Docker 镜像未构建/未推送 ghcr 的修复（commit 28edbe4，v1.2.0 已强制移到该提交）
- 现象：release workflow 跑完但 ghcr 里没有镜像，release 整体仍绿（用户报告"docker 没构建也没推 ghcr"）。
- 根因：`.dockerignore` 排除了 `frontend/dist`；release 的 docker job 在 runner 本地预构建前端后由 `Dockerfile.release` 的 `COPY frontend/dist` 把产物塞进 scratch 镜像，构建上下文受 .dockerignore 约束导致该 COPY 失败。且 docker job 原挂 `continue-on-error: true`，失败被静默吞掉。
- 修复：
  1. `.dockerignore` 去掉 `frontend/dist` 排除（主多阶段 Dockerfile 内部会重建 dist，含陈旧的 dist 无害），加注释说明。
  2. 去掉 docker job 的 `continue-on-error`，让 docker 失败变红可见。
  3. `release.yml` 顶层 `permissions` 补 `actions: write`（`cache-to: type=gha` 需要，否则 gha 缓存导出因 token 权限不足失败）。
  4. release 步骤加 `overwrite: true`，同一 tag 重推可幂等重建 Release（避免 "release already exists"）。
- docker job 还在本地 `go build -o ../warhutv`，二进制在仓库根；Dockerfile.release 的 `COPY warhutv` / `COPY frontend/dist` / `COPY data/config.example.json` 均依赖构建上下文 `.` 含这些产物（warhutv 与 config.example.json 本就未被忽略）。
- 验证：`git push --force origin v1.2.0` 重新触发 release.yml；去 Actions 看 docker job 是否绿 + ghcr 是否出现 `ououmm/warhutv:1.2.0` 与 `:latest`。
- 教训：`.dockerignore` 会影响所有以仓库根为上下文的 docker 构建，排除构建产物会破坏"本地预构建 + 拷贝进 scratch"的 release 模式；调试 CI 时 `continue-on-error` 会掩盖失败。

## GHCR 推送 write_package 被拒（commit 7423d2c）
- 现象：docker 构建成功，但 `docker build-push` push 到 `ghcr.io/ououmm/warhutv` 报 `denied: permission_denied: write_package`（用户旧日志里是 `warhuttv` 双 t，来自更早的旧配置；当前 metadata-action 推的是单 t `warhutv`，源自 repo slug `OuOumm/WarHutTV` 小写）。
- 根因：登录用 `GITHUB_TOKEN`，它是**仓库作用域**令牌，只能写"归当前仓库所有"的包。若目标包已存在且当初由别的上下文（本地手动 push / 别的仓库 / 早期配置）创建，则本仓库 GITHUB_TOKEN 推送被拒。顶层 `permissions: packages: write` 对"包已被别的身份占有"无能为力。
- 修复（workflow 层，commit 7423d2c）：
  1. docker job 显式 `permissions: contents: read, packages: write`。
  2. 登录改为优先 PAT 密钥 `GHCR_TOKEN`（`if: secrets.GHCR_TOKEN != ''`），无则回退 `GITHUB_TOKEN`（`if: secrets.GHCR_TOKEN == ''`）。PAT 带 `write:packages` 作用域可写账户下任意包，绕开包归属限制。
- 用户已选路线 B：在 GitHub 账号 Packages 页删除了已有 `warhutv`/`warhuttv` 包。已将 `v1.2.0` 强制移到 `7423d2c` 并重推触发 release.yml（docker job 走 GITHUB_TOKEN 回退，全新创建并占有包）。待用户回 Actions 确认 docker job 绿 + ghcr 出现镜像。
  - 路线 A（推荐）：建 classic/fine-grained PAT 勾 `write:packages`，在仓库 Settings→Secrets 建 `GHCR_TOKEN` 后我重打 tag。
  - 路线 B：GitHub 账号 Packages 页删掉已有 `warhutv`/`warhuttv` 包，本仓库 GITHUB_TOKEN 重新创建并占有后即可推。
- 教训：GHCR `write_package` 被拒多为"包已被别的身份占有"，GITHUB_TOKEN 救不了；标准解法是 PAT（write:packages）或删除已有包由本仓库重建。

## release.yml workflow 文件解析失败（commit 6bf9c43，已强制移到 v1.2.0）
- 现象：GitHub 报 `Invalid workflow file: Unrecognized named-value: 'secrets'`，指向 `if: ${{ secrets.GHCR_TOKEN != '' }}` 与 `if: ${{ secrets.GHCR_TOKEN == '' }}` 两行。
- 根因：GitHub Actions **禁止在 job/step 的 `if:` 条件里使用 `secrets` 上下文**（安全限制），只能在 `with:`/`env:` 等字段里用。
- 修复：合并两个分支登录步骤为单一 `Login to GHCR` 步骤，在 `with:` 的 `password` 字段做回退：`password: ${{ secrets.GHCR_TOKEN != '' && secrets.GHCR_TOKEN || secrets.GITHUB_TOKEN }}`。逻辑不变（有 PAT 用 PAT，无则 GITHUB_TOKEN），但不再用 `if` 引用 secrets。
- 验证：`grep -nE "if:.*secrets" .github/workflows/*.yml` 无输出即合格。已将 v1.2.0 强制移到 6bf9c43 重推触发 release.yml。
- 教训：**GitHub Actions 的 `if:` 条件不能用 `secrets` 上下文**；要做"有密钥/无密钥"分支，应在 `with:`/`env:` 里用表达式回退（如 `secrets.X != '' && secrets.X || secrets.Y`），或改用 `vars` 先判断。
