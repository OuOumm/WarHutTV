## 修复换源后续播从头开始（已提交 cfa1044，1 文件 +12/−1）
- 现象：换源后 Toast 提示「从 22:30 继续播放」，但实际从头播。
- 根因：换源时进度保存 interval（每 500ms 调 handleTimeUpdate）在新视频加载瞬间把 currentTime=0 写入 currentTime/seekTimeRef，并 `historyStore.record(progress:0)` 污染续播点；导致新视频 loadedmetadata 时 `seekTimeRef>0` 不成立而不 seek，从头播。
- 修复：新增 `seekingRef`（仅换源→seek 完成之间为 true），`saveInterval` 加 `!seekingRef.current` 守卫暂停进度回写；seek 完成或 `video:canplay` 解除锁。初始加载不受影响（初始 seekingRef 为 false）。
- 验证：tsc -b 0 错误、vitest 29/29、vite build 成功。

## 播放记录机制重构（方案A，团队 SOP：software-enhance-playrec）
- 用户问"有没有更安全更简洁的播放记录办法？换源从上一个源的记录继续、成功播放后实时记录"。结论是：续播底层已具备（key 源无关 + P1-1 episodeIndex + b65027c livePos），真正要升级的是"记录机制"。选**方案A**：3s 节流 + 事件刷写，而非 500ms 直写。
- 5 项目标落地（工程师 `IS_PASS: YES`，QA 独立回归 **56/56 通过**，路由 `NoOne` 源码无需返工）：
  1. **单一写器**：`usePlaybackProgress` 重写为仅 `{tick, flush, clearInvalidHistory}`，全部落库集中于此；Player 只喂 `(time, duration?)` + `onFlush`。
  2. **节流+事件刷写**：`THROTTLE_MS=3000`；`tick` 按 `lastWriteRef` 时间门控（否则置 `dirtyRef`）；挂载 `visibilitychange(hidden)/pagehide/beforeunload` 监听 dirty 时立即 `flush`。较原 500ms 直写约少 6 倍 I/O，关页不丢。
  3. **修复 duration 死字段**：`PlayState` 加 `duration`；Player 上报真实 `video.duration`（NaN/Infinity/0→undefined）；`tick` 仅有效 duration 时 patch；`historyStore.record` 写真实值，"继续观看"卡片可显真实进度%。
  4. **门控到成功播放**：`art.on('play')` → `art.on('video:playing')`（grep 确认 artplayer 有该事件）；`flush` 门槛 `if (!st.currentTime || !st.duration) return;`，任一为 0 不落库——卡顿/失败源不污染历史。
  5. **换源前 flush**：`useSourceSwitch.handleSourceSwitch` 在 same-source 守卫后调 `deps.progressFlush?.()`，锁死"换源继续上一个源记录"（写同一条 `site:vod_id`，新源 applyResumeProgress 读同一记录续播）。

## 修复换源后「从0播放」复发（Player.tsx 换源 seek 加固）
- 现象（用户复现）：换源后 toast 提示「从 XX:XX 继续播放」正确，但视频加载成功后实际从 0 开始播。
- 根因：此前 `resolveSwitchSeekTarget` 只修了**目标计算**（用状态内 seekTimeRef 而非视频元素，消除 0 竞态），但**seek 落地**仍不可靠。换源 switch effect 仅注册一次 `loadedmetadata` + `canplay` 的 `once` seek，且 `canplay` 处理器**无条件** `seekingRef.current = false`。当首次 seek 因 HLS 可搜索区间未就绪 / autoplay 抢跑而未真正落地时，`canplay` 已把 `seekingRef` 清零 → `saveInterval` 把新源起始 0 写回 `currentTime`/`seekTimeRef` 并污染历史 → 视频停在 0 再无后续 seek。
- 修复（`Player.tsx` 换源 effect，单文件）：
  1. 新增 `seekNetRef` 句柄，换源/卸载时 `art.off('video:timeupdate', net)` 清理，防泄漏。
  2. 抽出 `safeTarget()`（目标超新源时长则钳制到 duration）。
  3. `attemptSeek()` 发一次 `art.seek = safeTarget()`（暂不可 seek 时静默失败）。
  4. `video:seeked` 处理器**仅在 `|currentTime - target| ≤ 2s` 才**确认 seek 落地：置 `seekDoneRef`/`seekingRef=false` 并移除时间守卫——避免"no-op seeked"（currentTime≈0）被误判成功。
  5. `loadedmetadata`/`canplay`/`seeked` 三事件均触发 attemptSeek；再加 `video:timeupdate` 守卫：seek 未落地且实际位置落后目标 >2s 就反复重试，直到真正到达目标。**`seekingRef` 只在确认落地后才解除**，期间进度回写继续被挡，杜绝 0 回写。
  6. 无续播点（target=0）分支直接 `seekingRef=false`，新源从头播、进度正常回写。
- 验证：受管 node 直跑 `tsc -p tsconfig.app.json --noEmit` 0 错误；vitest `src/components/Player.seek.test.ts`(8) + `src/pages/play`(28) 全过。改动仅在工作区、未提交。
- 注：初始加载（create effect）的 seek 未同步加固——用户仅报换源场景，首屏续播此前未复现，先聚焦换源；如首屏也发现从头播可同样套用时间守卫。

## CORS 拦截（m3u8/ts 直连源站）议题终止 —— 用户决定不处理
- 现象：前端 `fetch` 第三方 CDN 的 m3u8（`utils/adblock.ts` 的 `fetchAndFilterM3U8`，主列表+子 m3u8）+ hls.js 在 Chrome 下用 XHR 拉 .ts 分片，源站无 `Access-Control-Allow-Origin` 头 → `blocked by CORS policy`（实测例：`v10.ppqrrs.com/.../index.m3u8`）。后端 `proxy.go` 仅代理数据源 JSON，无视频流代理。
- 探查结论（curl 实测）：该 CDN 返回 **403 Forbidden**（openresty，强防盗链 + URL 过期/token 机制），带 Referer+UA 仍 403。→ 仅代理 m3u8 不代理 ts 不可行（.ts 分片同 CDN 保护，hls.js 直连仍 403/CORS 挂首片）；全量流式代理虽可行但≈自建 CDN、带宽代价过大、性价比差。
- **决策：源站侧问题，前端/自建代理无解，放弃处理，不组队、不改代码。** 后续若切到能正常跨域/无强防盗链的源站，此议题可重新评估（再议按源可选代理 or 继续直连）。
