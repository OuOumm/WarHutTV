# 修复播放页换源进度丢失

## Goal

用户在播放页切换播放源时，新源应从当前源的最新有效播放位置继续播放，而不是从 `0:00` 或旧的历史进度开始。

## Background

- 播放进度由 `usePlaybackProgress` 异步写入 `historyStore`，换源流程随后通过 `applyResumeProgress` 读取该记录。
- 当前 `useSourceSwitch.handleSourceSwitch` 会调用异步 `progressFlush`，但未等待写入完成便继续读取历史记录，存在明确的“写后立即读”竞态。
- 当前工作区已有播放进度单写器、换源定位及剧集索引相关改动；本任务只补齐换源前的持久化同步保证，不覆盖或回退这些未提交改动。

## Requirements

1. 切换到不同播放源前，必须等待当前源最新有效进度完成持久化，再读取续播记录和应用新源。
2. 换源后继续使用当前剧集对应的最新有效时间；不得因异步写入尚未完成而回退到 `0:00`。
3. 点击当前已选源时维持现有无操作行为，不产生额外进度写入。
4. 保持现有换源失败、并发请求顺序、源选择状态及无有效进度时从头播放的行为不变。
5. 添加针对异步进度写入竞态的回归测试，测试必须能证明换源流程会等待写入完成。

## Acceptance Criteria

- [x] 在非零进度播放时切换到其他可用源，新源从切换前的最新有效播放位置继续，而非从 `0:00` 开始。
- [x] 当 `progressFlush` 是延迟完成的 Promise 时，换源流程在 Promise 完成前不会开始读取/应用续播数据。
- [x] 点击当前源不会调用 `progressFlush`，既有行为保持不变。
- [x] 无有效播放进度时仍按现有规则从头播放。
- [x] 相关 Vitest 回归测试通过。
- [x] 前端类型检查或构建检查通过。

## Out of Scope

- 重构播放器或历史记录整体架构。
- 修改换源搜索、去广告、自动连播或 UI 样式。
- 清理当前工作区中与本缺陷无关的未提交改动。
