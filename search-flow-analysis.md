# 搜索→播放流程分析与优化建议

## 当前流程

```
搜索页 (Search.tsx)
  ↓ SSE 流式搜索所有源，实时展示结果
  ↓ 用户点击视频卡片 → 跳转 /play/{site_key}/{vod_id}
  ↓
播放页 (Play.tsx)
  ① loadDetail() — 根据 URL 中的 site 和 id 加载当前源的详情
  ② loadDetail 完成后自动调用 startOptimize(videoName)
     ②-a 尝试从 sessionStorage 读取搜索页的缓存结果
     ②-b 缓存 miss → 再次发起完整 SSE 流式搜索（第二次！）
     ②-c 构建源列表（所有找到的源）
     ②-d 并发测速所有源：每个源取详情 → 取第一集 m3u8 → 测速
     ②-e 选出最快源 → 更新详情/集数 → 开始播放
```

## 问题总结

### 1. 重复搜索（最严重）

- 搜索页已经完成了一次完整的 SSE 流式搜索
- 播放页通过 `sessionStorage` 缓存复用结果，但 key 用 `search_results:{title}` 拼接
- 标题有空格差异、特殊字符、或精确匹配筛选后名称变化时，**缓存必然 miss**
- miss 后播放页会再发起一次完整的 SSE 搜索，用户体感是"搜了两遍"

### 2. 两层 Loading 叠加，等待时间过长

```
第一层：SearchingOverlay（正在搜索播放源）  — 搜索所有源
第二层：OptimizingOverlay（正在优选最佳播放地址） — 对每个源测速
```

两层串行执行，总等待时间 = 搜索耗时 + 所有源测速耗时。用户从点击到看到画面可能需要 10~20 秒。

### 3. 播放页职责过重

Play.tsx 承担了三个独立职责：

- **搜索管理**：SSE 流式搜索、进度展示
- **测速管理**：并发测速所有源、展示测速动画
- **播放管理**：播放器控制、集数切换、历史记录

一个 500+ 行的组件里混杂了大量与播放无关的网络请求和状态管理。

### 4. 测速粒度过粗

每个源的测速流程：

```
构建源列表 → 对每个源调用 getCachedDetail() → 取 vod_play_url 的第一集 → 测 m3u8 速度
```

但搜索结果中已经有 `vod_id`，可以直接从搜索结果获取播放地址做测速，无需再额外请求详情。

### 5. 搜索结果丢失/不完整

播放页的 `startOptimize` 在构建源列表时做了二次筛选（精确匹配 + 黄色内容过滤），可能过滤掉搜索页原本展示的某些源，导致可用源变少。

---

## 优化建议

### 方案：搜索阶段顺带测速，播放页只负责播放

**核心思路**：把"搜索所有源"和"测速"这两步合并，搜索完成后每个源已经带上了速度信息，播放页直接播放，不再阻塞。

```
搜索页：SSE 流式搜索 + 每个源返回时顺便测首集速度（前端并发测速）
         ↓ 结果已带速度标记（loadSpeed / pingTime）
播放页：点击即播，不再触发 search + optimize 流程
         ↓ 可选：后台静默检查是否有更快源，有则 toast 提示
```

### 具体改动点

| # | 改动 | 说明 |
|---|------|------|
| 1 | **搜索结果携带速度信息** | 每个 source result 的 `done` 事件中附带该源首集 m3u8 的响应耗时；或前端收到结果后立即对首集做轻量测速 |
| 2 | **Play 页去掉 `startOptimize`** | 点击即播，不再自动触发搜索+测速流程；用户点击时直接使用 URL 中的 site + id 加载详情并播放 |
| 3 | **sessionStorage 缓存 key 改用 vod_id 组合** | 用 `site_key:vod_id` 作 key，不再依赖标题字符串，命中率大幅提升 |
| 4 | **测速改为后台可选** | 播放后在后台静默测其他源，发现更快的只 toast 提示"发现更快源，是否切换？"，不阻塞播放 |
| 5 | **搜索页增加"源数"标签** | 每个视频卡片显示"X 个源可用"，帮助用户感知源的丰富度，引导用户选择源多的结果 |

### 预期效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 点击到播放 | 10~20 秒（搜索+测速） | < 3 秒（直接播放） |
| 网络请求次数 | 2 次 SSE + N 次详情 + N 次测速 | 1 次 SSE + N 次测速（搜索阶段完成） |
| 播放页复杂度 | ~500 行，3 个职责 | ~200 行，只管播放 |
| 缓存命中率 | 低（标题 key 不稳定） | 高（vod_id key 稳定） |

### 风险点

- 搜索页增加测速会延长搜索总时长 → 可做成**后台渐进式测速**，搜索先返回结果，速度信息逐步填充
- 测速需要消耗用户带宽 → 可设置为可选开关，或只对前 N 个源测速
